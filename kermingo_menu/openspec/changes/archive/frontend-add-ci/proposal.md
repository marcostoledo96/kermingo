# Change: frontend-add-ci

## Why
We have `pnpm lint` running clean (0/0) and `pnpm build` passing 14/14 pages. Without CI, regressions can land silently:
- A new file with `set-state-in-effect` won't be caught
- A new `console.log` or unused import won't be caught
- A breaking type change that only manifests in build won't be caught

The repo is on GitHub (`github.com/marcostoledo96/kermingo`). Adding a GitHub Actions workflow is the natural place to enforce quality gates.

## What changes
- **NEW** `.github/workflows/frontend-ci.yml` — GitHub Actions workflow that:
  - Triggers on `push` to `main` and on `pull_request` targeting `main`
  - Sets up Node 22 + pnpm 11 (the versions used locally)
  - Caches pnpm store for speed (keyed on lockfile hash)
  - Runs `pnpm install --frozen-lockfile` in `frontend/`
  - Runs `pnpm lint` — fails on any warning or error
  - Runs `pnpm build` — catches TypeScript and Next.js errors
- **NEW** `.nvmrc` — pins Node version for local dev consistency (matches what CI uses)
- No source code changes. No backend changes. No new dependencies.

## Why GitHub Actions
- The repo is on GitHub
- Free for public repos
- Native pnpm support via `pnpm/action-setup`
- Native caching via `setup-node`'s `cache: pnpm` flag
- Fast (typically 1-2 minutes for a frontend install + lint + build)

## Why pnpm 11 + Node 22
- Locally we use pnpm 11.5.1 and Node 22.22.1
- Both are LTS / stable
- Lockfile is generated by pnpm 11

## Impact
- 2 new files: `.github/workflows/frontend-ci.yml`, `.nvmrc`
- 0 source code changes
- Future PRs will run CI automatically

## Out of scope
- Backend CI (Jest/Supertest) — separate change
- Tests for the frontend (no test suite yet) — separate change
- Auto-deploy on merge to main — separate change (would need Vercel/deploy keys)
- Preview deploys per PR — separate change (Vercel integration)
- Multi-Node matrix (test on 20, 22) — overkill for an event project, can add later
- Dependabot / Renovate auto-PRs — separate change

## Risks
| Risk | Likelihood | Mitigation |
|------|------------|------------|
| CI takes too long (>5 min) | Low | Cache pnpm store; typical install + lint + build is 1-2 min |
| CI fails on legitimate code that doesn't lint cleanly in CI but does locally | Med | We just verified `pnpm lint` exits 0 locally with the same Node + pnpm versions. Risk is low. |
| pnpm version mismatch (locally 11.5.1, CI installs 11.x latest) | Low | Use `pnpm/action-setup@v4` with `version: 11` to pin to major version |
| GitHub Actions minutes usage | Low | Free for public repos; the workflow runs ~1-2 min per push/PR |
| pnpm install fails because of platform-specific binary (sharp, unrs-resolver) | Low | Already in `onlyBuiltDependencies` in `.npmrc` |

## Rollback
Delete `.github/workflows/frontend-ci.yml` and `.nvmrc`. The repo continues to work as before.

## Dependencies
- GitHub Actions (free for public repos)
- pnpm 11 (already used locally)
- Node 22 (already used locally)

## Success criteria
- [ ] `.github/workflows/frontend-ci.yml` exists and is valid
- [ ] `.nvmrc` exists with `22`
- [ ] Pushing to main (or opening a PR) triggers the workflow
- [ ] Workflow runs `pnpm install --frozen-lockfile` successfully
- [ ] Workflow runs `pnpm lint` and exits 0
- [ ] Workflow runs `pnpm build` and exits 0
- [ ] Total CI time < 5 minutes
- [ ] If anyone introduces a warning or build error, CI fails
